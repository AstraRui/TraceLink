# Модель безопасности

## Управление секретами

- **Ни один секрет не попадает в репозиторий.** Реальный `.env` — в `.gitignore`;
  отслеживается только `.env.example` с заглушками.
- Все чувствительные значения (пароль БД, ключ подписи JWT, внутренний токен,
  пароль админа) читаются из переменных окружения через `pydantic-settings`.
- Docker Compose прокидывает их через `env_file`, поэтому секреты **никогда не
  «запекаются» в образ** — они существуют только во время выполнения.
- `pre-commit` включает `detect-private-key`, а `ruff` помечает захардкоженные
  секреты — как вторая линия защиты.

## Аутентификация (админка)

- Пароли хэшируются **bcrypt** (`passlib`); открытый текст не хранится.
- При входе выдаётся **JWT** (`python-jose`, HS256) в **HttpOnly, SameSite=Lax**
  cookie:
  - `HttpOnly` → недоступна из JavaScript (защита от кражи токена через XSS);
  - `SameSite=Lax` → снижает риск CSRF;
  - `Secure` → включается через `COOKIE_SECURE=true` при отдаче по HTTPS.
- У токена есть срок жизни (`JWT_EXPIRE_MINUTES`); просроченные/некорректные
  токены отклоняются, пользователя перенаправляет на `/login`.
- Закрытые маршруты зависят от `get_current_user`; сбои поднимают
  `NotAuthenticatedError`, который логируется (**попытки неавторизованного
  доступа фиксируются для мониторинга**) и редиректит на вход. Неудачные попытки
  входа тоже логируются.

## Авторизация между сервисами

- Управляющие и разрушающие эндпоинты — `POST`/`GET`/`DELETE /api/links`
  (redirect-service) и `DELETE /api/clicks/{code}` (stats-service) — **не
  публичны**. Они требуют заголовок `X-Internal-Token`, который сравнивается в
  **постоянном времени** (`secrets.compare_digest`) против тайминг-атак.
- Админка подставляет этот токен (`INTERNAL_API_TOKEN`) во все такие вызовы.
- nginx выставляет наружу только UI админки и публичный редирект — внутренние API
  снаружи не маршрутизируются.

## Приватность

- IP посетителей **никогда не хранятся в открытом виде**. Они хэшируются SHA-256
  (`ip_hash`) перед записью; источник — `X-Forwarded-For` (его проставляет nginx).

## Трассировка и логи

- Идентификатор трассировки `X-Request-ID` — это случайный непредсказуемый id
  запроса, а не секрет. Он не содержит персональных данных.
- Логи в `system_logs` пишут сами сервисы; таблица доступна только во внутренней
  сети, а страница **Logs** в админке закрыта аутентификацией.

## Сетевая устойчивость / без обращений к третьим сторонам

- Фронтенд-библиотеки (`htmx`, `Chart.js`) и сборка Tailwind **лежат локально** —
  без зависимости от CDN. QR-коды генерируются **локально** через `qrcode`, без
  внешнего QR-API. Система полностью работоспособна офлайн.

## Чек-лист для продакшена

Это учебный/практический уровень. Для продакшена:

- [ ] Заменить `.env` на реальный менеджер секретов (Vault / Doppler / облачный).
- [ ] Включить `COOKIE_SECURE=true` и терминировать TLS на шлюзе.
- [ ] Ротация `JWT_SECRET`, `INTERNAL_API_TOKEN`, паролей БД и админа.
- [ ] Добавить rate limiting на `/login` и создание ссылок.
- [ ] Ограничить сетевой доступ к БД/Redis (здесь они уже только внутренние).
- [ ] Настроить ретеншн/ротацию таблицы `system_logs`.
