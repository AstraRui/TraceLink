# Руководство по возможностям

Здесь описаны продуктовые и UX-возможности админки помимо базового сценария
«сократить и перенаправить».

## Тёмная тема

- Tailwind настроен в режиме `darkMode: "class"`.
- Переключатель (солнце/луна) — в шапке (и на странице входа). Выбор сохраняется
  в `localStorage` и применяется **до первой отрисовки** маленьким inline-скриптом
  в `<head>`, поэтому нет «мигания» темы.
- У всех карточек, текста, рамок и полей есть `dark:`-варианты.
- **Графики реагируют на тему**: при переключении `charts.js` перекрашивает оси,
  сетку и легенды (без пересоздания графиков).

Реализация: `static/js/theme.js`, `templates/base.html`, `static/js/charts.js`.

## Статистика в реальном времени

Блок статистики обновляется без перезагрузки страницы. Чтобы избежать «прыжков»
вёрстки и мигания, графики **создаются один раз**, а данные обновляются на месте:

- htmx раз в 5 секунд запрашивает `GET /partials/stats.json` (только данные);
- скрипт вызывает `chart.update()` — графики плавно перерисовываются;
- KPI-строка (Всего / Сегодня / Топ-ссылка) обновляется тем же payload.

Графики: заливка-градиент со сглаживанием (`tension`) для кликов по дням и
кольцевые диаграммы по устройствам и браузерам. Контейнеры графиков имеют
`position: relative` и фиксированную высоту, поэтому диаграммы не «разъезжаются».

## Панель здоровья системы (System Health)

На дашборде — живые индикаторы состояния всех зависимостей:

```html
<div id="health-block" hx-get="/partials/health" hx-trigger="load, every 10s" hx-swap="innerHTML">
```

`GET /partials/health` выполняет четыре проверки **параллельно** (`asyncio.gather`):

| Индикатор | Проверка |
|---|---|
| PostgreSQL | `SELECT 1` через сессию БД админки |
| Redis | `PING` |
| Redirect API | `GET redirect-service/health` |
| Stats API | `GET stats-service/health` |

Каждая проверка защищена — любой сбой превращается в красную точку **down**, а не
в исключение. Зелёная точка — **up**.

Реализация: `app/services/health.py`, `templates/partials/health.html`.

## Каскадное удаление

У каждой карточки ссылки есть кнопка **Delete** (`hx-delete` + подтверждение).
Удаление каскадно проходит по сервисам:

```
admin  DELETE /links/{code}
  → redirect-service  DELETE /api/links/{code}   (удалить запись + очистить кэш Redis)
  → stats-service     DELETE /api/clicks/{code}   (удалить все клики этого кода)
```

Оба нижестоящих эндпоинта требуют заголовок `X-Internal-Token` и идемпотентны.
После успеха карточка убирается со страницы (htmx).

## Детальная страница ссылки

Клик по **Details** на карточке открывает `GET /links/{code}` — страницу со
статистикой только по этому короткому коду (свои KPI и графики) и кнопкой
удаления. Неизвестный код перенаправляет обратно на дашборд.

## Подсказка про «Private»

Рядом с чекбоксом создания и на бейдже карточки есть иконка «ⓘ» с подсказкой.
Текст честно объясняет логику: короткая ссылка **всё равно** редиректит любого,
у кого она есть; флаг «приватная» лишь помечает ссылку, чтобы исключить её из
публичных списков.

## Сквозная трассировка и логи (Distributed Tracing)

Каждый запрос получает идентификатор трассировки **`X-Request-ID`**:

- nginx генерирует его (или использует переданный клиентом) и передаёт во все
  сервисы;
- в каждом сервисе middleware кладёт его в `contextvar`, а логгер добавляет
  `trace_id`, уровень и имя сервиса в каждую запись;
- идентификатор пробрасывается дальше по внутренним HTTP-вызовам
  (admin → redirect/stats, redirect → stats при отправке клика).

Все три сервиса **асинхронно** пишут логи в общую таблицу `system_logs`
(PostgreSQL): запись попадает в очередь `asyncio.Queue`, фоновая задача вставляет
её в БД, не задерживая обработку запроса.

### Просмотр логов

Кнопка **Logs** в шапке открывает `GET /logs`:

- фильтры по сервису (Admin / Redirect / Stats) и уровню (INFO / WARNING / ERROR);
- автообновление каждые 5 секунд (htmx);
- цветовая индикация уровней (ERROR — красный, WARNING — янтарный, INFO — синий);
- кнопка копирования полного `trace_id`, чтобы пройти путь одного запроса по всем
  микросервисам.

Реализация: `app/core/logging.py` (общий модуль во всех сервисах),
`app/services/logs.py`, `templates/logs.html`.

## Документация API (Swagger) через шлюз

Swagger UI каждого сервиса доступен через шлюз:

- Redirect: <http://localhost:8080/api/redirect/docs>
- Stats: <http://localhost:8080/api/stats/docs>

Каждый сервис выставляет свои `docs_url` / `openapi_url` под префиксом шлюза
(переменная `DOCS_PREFIX`), а nginx проксирует именно эти пути. Сырые схемы —
по `/api/redirect/openapi.json` и `/api/stats/openapi.json`.

> Примечание: используется стандартный Swagger UI FastAPI, который подгружает свои
> JS/CSS с CDN. Сами схемы (`openapi.json`) — полностью локальные.

## Маршрутизация шлюза (nginx)

| Путь | Куда |
|---|---|
| `/`, `/login`, `/logout`, `/health`, `/logs` | админка |
| `/static/…`, `/partials/…`, `/links`, `/links/…` | админка |
| `/api/redirect/docs`, `/api/stats/docs` (+ `openapi.json`) | соответствующий сервис |
| `/{code}` (ровно 6–8 url-safe символов) | redirect-service |
| всё остальное | админка |

nginx использует встроенный DNS Docker (`resolver 127.0.0.11`) и переменные в
`proxy_pass`, поэтому повторно резолвит имена сервисов на каждый запрос — и
маршрутизация не ломается после пересборки контейнеров (когда меняются их IP).
